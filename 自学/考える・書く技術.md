# 第一章 なぜピラミッド構造なのか

## ピラミッド型へ並び替える
- ピラミッドの思考パターンが以下の頭の中の要求を満たす
    - マジックナンバー7でストップせよ
    - 関連付けの論理を明らかにせよ
### マジックナンバー7
- 人間が同時に覚えられるものの数の限界は7
### 論理を述べる必要性
- 効率的に事象を伝えるためには、まずグループ化した姿を伝え、その後で、それぞれの事象を伝えていく必要がある

## トップダウンに配列する
- 読み手はトップダウンに考えを記憶しようとするため、トップダウンで書き手の考えが示されていれば、読み手はより容易に理解できるようになる

## ボトムアップで考える
- 各文章をグループ化して段落を作り、さらに各段落をグループ化することで章を作り、最後に章をまとめて文書を完成させる
- 適切な文書構成であるかどうかは、それぞれのメッセージ関係が正しいピラミッド型になっているかどうかでチェックする
- 以下の3つの鉄則を文書作成前にチェックする
    1. どのレベルであれ、メッセージはその下位グループ群を要約するものである
    2. 各グループのメッセージは、常に同じ種類である
    3. 各グループのメッセージは、常に論理的に順序付けられている
- わかりやすい文書を書くキーポイントは、筆をとる前にまず各々のメッセージをピラミッド型に並べてみて、3つの鉄則と照らし合わせてみること

# 第二章 ピラミッドの内部構造
- 自分の考えを発見するための、ピラミッドの厳格な構造フレーム
    - 主ポイントと補助ポイント間の縦の関係
    - 補助ポイント同士の横の関係
    - 導入部のストーリー展開

## 縦の関係
- 読み手の関心を維持する方法は、答えを準備していないことに対して疑問を起こさせないことであり、疑問がわく前に答えを言うことも避けるべき

## 横の関係
- 演繹法に書くのであれば、第二のポイントで第一のポイントの主部か述部について述べ、第三のポイントで先の二つのポイントから「それゆえに」という結論を引き出すという形で議論を展開する
- 帰納的に書くのであれば、グループ内の考えは論理的に同じものでなければならず、同類の内容を表す事柄でくくられる

# 第四章 導入部の構成
## コンサルティングの場合
- コンサル文書は、文書自体が長いという点及び主として行動を促す目的で書かれるという点で、一般のビジネス文書と異なる
### 提案書
- 以下のアプローチを用いる
    1. 状況     = あなたは問題を抱えている（問題状況を説明する文章を一つか二つ）
    2. 複雑化   = その問題を解決するために、第三者へ打診することを決めた
    3. 疑問     = 我々は、問題解決のためにあなたが雇うべき第三者であるか？
- 疑問に対する答えは以下の四つのパートで構成される
    1. 我々は問題を理解している
    2. 我々は問題解決に適したアプローチを持っている
    3. 我々はこのアプローチを実施するにあたり、十分な経験を有している
    4. 我々の価格条件は納得のいくものである
- コンサルタントが顧客をよく知っている場合や、提案書が単なる形式を満たすためのものである場合、導入句において直接的に問題の記述を行うべき
    1. 状況     = 御社は問題を抱えている（３，４つの段落で説明）
    2. 複雑化   = これを解決するためにコンサルタントの力を借りたい
    3. 疑問     = 我々は、どのようにして問題解決に力を貸すか？
- 以上の場合は、問題解決に向かってのアプローチの取り方を中心に記述する。理論的には、顧客がこのコンサルタントを起用するかどうかを決定するキーポイントはこのアプローチの考え方の部分である

### 進捗状況報告書
- 第一フェーズ終了後の進捗状況報告の構成は常に以下のようになる
    1. 状況     = 我々はXという問題について作業してきた
    2. 複雑化   = 我々は、提案書の中で、分析第一ステップの目的はYが当該ケースかどうかを判断することであると説明して。我々は今その作業を完了した
    3. 疑問     = 我々は何を発見したか？
- 次の進捗状況報告の際には、以下のような形式となる
    1. 状況     = 先の進捗状況の報告で、あなたの会社は生産能力の過剰問題を抱えていると伝えた
    2. 複雑化   = あなたは、競合相手がこのビジネスから撤退しそうであるため、これは長期的な問題とはならないと考えた。あなたは、この考えの妥当性について我々に調査を依頼し、我々はこの調査を終了した
    3. 疑問     = 何を発見した？
    4. 答え     = 我々が発見したことは、やはり生産能力過剰問題を抱えていて、考えていた以上に深刻であること

# 第八章 問題を定義する
- 問題を解決するということは、論理的に関連付けられた以下の質問群に対して答えを見つけるという作業を伴う
    1. 問題または改善の機会がありそうか
    2. 問題はどこにあるのか
    3. 問題はなぜ存在するのか
    4. 問題に対し何ができるか
    5. 問題に対し何をすべきか

## 問題定義のフレームワーク
- 現在の結果と期待にはギャップが存在していて、そのギャップが問題である
- 以下の3つの質問に答える必要がある
    1. 今何が起きているか
    2. 今の何が好ましくないのか（R1）
    3. 代わりに何を望んでいるのか（R2）

## 問題を配置する
- 解決策を生み出すためには以下の4つの要素を特定する必要がある
    1. スタートポイント/オープニング（問題発生時の状況）
    2. 懸念される出来事（外的/内的引き金・新たな認識）
    3. R1（望ましくない結果）
    4. R2（望ましい結果）
- コンサルティングの提案書を描く場合など、情報が不十分なために、なぜ問題が存在すると考えるのか、その根拠を具体的に明らかにすることができない場合も多くあるが、読み手が自分の構造やプロセスで何を不満に思っているかを明らかにできるはず。この場合、懸念される出来事をすっ飛ばしてR1へ進む

## 疑問を見出す
- 書き手の大きな間違いの要因は、その問題を解決するために読み手がすでに何らかの行動をとったかどうかをはっきりとさせないことにある
- 状況・R1・R2を状況と複雑化のどれに入れればよいかが問題定義の勝負

# 問題分析を構造化する
- 調査者は調査に取り掛かる前に、初期のデータ収集作業の方法をよく考えて構造化し、論理的に首尾一貫した調査結果が得られるように工夫する必要がある

## データ収集から始める
- データ収集前に以下のアプローチで問題分析を構造化する
    1. いくつかの仮説を設定する
    2. 誤った仮説を捨て、正しい仮説を選定できるように、仮説の実験装置（判断するためのチェックポイント）を考案する
    3. 仮説を証明する明確な結果が得られるまで実験を繰り返す
    4. 証明された仮説に基づき、望ましい行動を提案する
